# Deployment Runbook — Vraman Holidays

Two isolated environments:

| Environment | Branch        | Host                          | Database (Neon branch)        |
| ----------- | ------------- | ----------------------------- | ----------------------------- |
| **Dev / staging** | `development` | **Vercel** (auto-deploy)      | Neon **dev** branch           |
| **Production**    | `production`  | **Babal cPanel** (Passenger)  | Neon **production** branch    |

The app is Next.js 16 (standalone output). PostgreSQL stays on **Neon** (the cPanel
host has no native Postgres). The production build runs in **GitHub Actions** — never
on the shared host — and the built bundle is published to a `babal-deploy` branch that
cPanel Git pulls.

```
feature/* ──merge──▶ development ──push──▶ Vercel (dev DB)
                          │
                       merge
                          ▼
                     production ──push──▶ GitHub Actions build
                                               │ force-push built bundle
                                               ▼
                                          babal-deploy branch
                                               │ cPanel "Deploy HEAD Commit"
                                               ▼
                                  Babal cPanel + Passenger (prod DB)
```

> `babal-deploy` is **generated by CI** — never edit or commit to it by hand.

---

## Branch workflow

1. Branch from `development`: `git checkout development && git checkout -b feature/x`.
2. Open a PR into `development`. Merging deploys to **Vercel** (dev DB) for testing.
3. When dev looks good, merge `development` → `production`:
   ```bash
   git checkout production && git merge --ff-only development && git push origin production
   ```
4. The push to `production` triggers the CI build and updates `babal-deploy`.
5. In cPanel, deploy `babal-deploy` (see below). **Smoke-test on the live domain.**

## Environment variable matrix

Set the real values in each place (templates: [.env.example](.env.example)).

| Variable | Vercel (dev) | GitHub Actions (build) | cPanel Node App (prod) |
| --- | --- | --- | --- |
| `DATABASE_URL` | Neon **dev** string | Neon **prod** string¹ | Neon **prod** string |
| `AUTH_SECRET` / `NEXTAUTH_SECRET` | dev secret | — | prod secret |
| `AUTH_TRUST_HOST` | `true` | — | `true` |
| `NEXTAUTH_URL` / `AUTH_URL` | Vercel URL | — | `https://vramanholidays.com.np` |
| `NEXT_PUBLIC_APP_URL` | Vercel URL | prod URL² | `https://vramanholidays.com.np` |
| `PAYMENTS_MODE` | `sandbox` | — | `production` |
| `ESEWA_*`, `KHALTI_SECRET_KEY`, `STRIPE_SECRET_KEY`, `STRIPE_WEBHOOK_SECRET` | sandbox keys | — | production keys |
| `RESEND_API_KEY` | dev/test key | — | prod key |
| `CLOUDINARY_URL`, `CLOUDINARY_API_KEY`, `CLOUDINARY_API_SECRET` | yes | — | yes |
| `CLOUDINARY_CLOUD_NAME` | yes | **yes**² | yes |
| `NEXT_PUBLIC_GA_ID` | yes | **yes**² | yes |

¹ The build runs `generateStaticParams()` (blog/destinations/packages/legal), which
queries the DB — so CI needs a working `DATABASE_URL`. Using the prod branch is fine
(read-only at build).
² `NEXT_PUBLIC_*` and `CLOUDINARY_CLOUD_NAME` are **inlined at build time**, so they
must be present in **GitHub Actions secrets**, not just on the host.

**Runtime-only** secrets (`AUTH_SECRET`, payment keys, `RESEND_API_KEY`, etc.) are
**not** needed by CI — set them only in the cPanel Node App panel.

## One-time setup

### 1. Neon
Two branches, already provisioned (both fully seeded with the same content):

| Role | Endpoint host | Used by |
| --- | --- | --- |
| **Production** | `ep-dry-glade-ao6mhn23` (0 test bookings, schema up to date) | cPanel + CI build |
| **Dev** | `ep-floral-band-ao52b9z6` (existing) | Vercel + local `.env` |

**Pooled vs direct endpoint (important):**
- **App runtime** `DATABASE_URL` (cPanel, CI build, Vercel) → use the **pooled**
  string (`...-pooler...`, keep `sslmode=require&channel_binding=require`).
- **Migrations** (`prisma migrate deploy`) → use the **direct** string (same host
  **without** `-pooler`). Pooled/PgBouncer connections break DDL.

The production branch already has all 3 migrations applied, so no migrate/seed is
needed now. For future schema changes, run `prisma migrate deploy` against the
**direct** prod endpoint before deploying code that depends on them.

### 2. Vercel (dev environment)
- Project → **Settings → Git → Production Branch = `development`**.
- Project → **Settings → Environment Variables**: add the "Vercel (dev)" column above
  (point `DATABASE_URL` at the Neon **dev** branch).
- Push to `development` → confirm the Vercel deploy hits the dev DB.

### 3. GitHub Actions secrets
Repo → **Settings → Secrets and variables → Actions** → add:
`DATABASE_URL` (prod), `NEXT_PUBLIC_APP_URL`, `CLOUDINARY_CLOUD_NAME`,
`NEXT_PUBLIC_GA_ID`, `NEXTAUTH_SECRET`. (The workflow uses the built-in
`GITHUB_TOKEN` to push `babal-deploy`; no extra token needed.)

### 4. cPanel — Node.js App (Passenger)
- **Setup Node.js App → Create Application**
  - Node version: **22.22.2**
  - Application root: `vraman-app` → absolute path `/home/vramanho/vraman-app`
    (must match `DEPLOYDIR` in [.cpanel.yml](.cpanel.yml))
  - Application URL: the domain root
  - **Startup file: `passenger.js`** (wrapper; falls back to `server.js`)
- Add all "cPanel Node App (prod)" env vars from the matrix.
- Do **NOT** click "Run NPM Install" — the standalone bundle ships its own
  `node_modules`. Running install would fight the bundle.

### 5. cPanel — Git Version Control
- **Create** a clone of the GitHub repo using a **read-only deploy key**:
  - In cPanel → Git Version Control → Create, paste the repo SSH URL.
  - cPanel shows an SSH key — add it to GitHub → repo **Settings → Deploy keys**
    (read access is enough).
  - Set the checked-out branch to **`babal-deploy`**.
- To deploy: **Manage → Pull or Deploy → "Update from Remote"** then
  **"Deploy HEAD Commit"**. cPanel runs [.cpanel.yml](.cpanel.yml), which rsyncs the
  bundle into `/home/vramanho/vraman-app` and touches `tmp/restart.txt` to restart
  Passenger.

### 6. SSL
- cPanel → **SSL/TLS Status** → run **AutoSSL** for `vramanholidays.com.np`
  (and `www`). Requires DNS already pointing the domain at this host/app —
  see the DNS notes at the bottom.

## Routine production deploy (after setup)

1. `git checkout production && git merge --ff-only development && git push origin production`
2. Watch the **Actions** tab → "Deploy to Babal (production)" goes green
   (it force-pushes to `babal-deploy`).
3. cPanel → Git Version Control → **Update from Remote** → **Deploy HEAD Commit**.
4. Run the **smoke test** below on `https://vramanholidays.com.np`.

If a deploy looks wrong, re-deploy the previous good commit from cPanel, or revert the
merge on `production` and push again.

## Post-deploy smoke test (run on the LIVE Babal site)

> Vercel and Passenger run Next.js differently (serverless vs a long-lived Node
> process). Passing on Vercel does **not** guarantee prod — verify here directly.

- [ ] `https://vramanholidays.com.np` — homepage loads with hero + sections, over HTTPS.
- [ ] A **package detail** page renders (itinerary, gallery, price).
- [ ] **Admin login** works at `/admin/login`; dashboard loads (RBAC intact).
- [ ] Create a **booking** through the flow → confirmation page + email received.
- [ ] **Promo code** applies and reduces the charged total.
- [ ] **Payment callbacks resolve**: eSewa, Khalti, and Stripe each return to the site
      and mark the booking paid (`/api/webhooks/stripe` reachable; Stripe webhook
      configured to the prod URL).
- [ ] Check the Node App log in cPanel for errors (esp. Prisma / DB connectivity).

## Things flagged / verify on first deploy

- **Prisma in the standalone bundle.** [next.config.ts](next.config.ts) sets
  `output: "standalone"` and force-includes `./src/generated/prisma/**` via
  `outputFileTracingIncludes`. The app uses the `@prisma/adapter-pg` driver adapter, so
  it talks to Neon over `pg` and typically needs **no native query-engine binary** at
  runtime. **If** the prod Node App log shows a missing Prisma engine / `.wasm` / a
  `Cannot find module '@/generated/prisma'` error, widen that glob (or add a copy step
  in the workflow) and redeploy.
- **Passenger host binding.** [deploy/passenger.js](deploy/passenger.js) forces
  `HOSTNAME=0.0.0.0` before loading `server.js` (Passenger otherwise sets a
  non-bindable machine name). If the app won't boot, try setting the startup file to
  `server.js` directly, or add `PORT`/`HOSTNAME` in the Node App env.
- **App directory path.** `DEPLOYDIR=/home/vramanho/vraman-app` in
  [.cpanel.yml](.cpanel.yml) must match the cPanel Node App "Application root". Change
  both together if you use a different folder.
- **next/image optimization** uses `sharp` on the Node server. Images already come from
  Cloudinary; if on-the-fly optimization misbehaves on the shared host, set
  `images.unoptimized: true` in next.config or use a Cloudinary loader.

## Alternative: rsync-over-SSH (if SSH is enabled)
Instead of the `babal-deploy` branch + cPanel Git pull, CI can `rsync` the assembled
`.next/standalone/` straight to `/home/vramanho/vraman-app` over SSH and then
`ssh … touch tmp/restart.txt`. It's fewer moving parts, but requires an SSH key in CI
and SSH access on the plan. The cPanel Git path is the documented default here because
it's the feature you asked to use.
